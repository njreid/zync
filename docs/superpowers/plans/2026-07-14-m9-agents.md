# M9 — Agents (human-gated): design for the M8-dependent remainder

> Status: DESIGN (2026-07-14). The M8-independent M9 work (metrics counters,
> op-log compaction + retention, quota, restore drill, agent-flow wire types +
> accept/reject UI with a stubbed producer) lands on `feat/m9-hardening`. This doc
> specifies the parts that need the M8 operators runtime (`feat/m8-operators`:
> `OperatorRuntime`, `LlmClient`, `OpIngestHook`) merged first, concretely enough
> that implementing them is mechanical. Semantics source of truth: the op/merge
> spec §8 (agents are human-gated, proposal-only, provenance-tagged writers).

## 1. Data model (op-log, no new tables)

Everything is nodes + fields in the existing op log; provenance comes from `Actor`
(core already has `Actor.Agent(id)`). Three node kinds:

- **Recommendation** (`kind="recommendation"`, authored `Actor.Operator(id)`):
  fields `agentKind` (which agent to run), `subjectId` (target node), `rationale`.
  Emitted by an operator whose typed output requests an agent (M8 runtime already
  validates typed outputs; a recommendation is just another emitted child node).
- **Agent task** (`kind="agent_task"`, authored `Actor.Human` — the approval IS the
  op): fields `agentKind`, `subjectId`, `recommendationId`, `status`
  (`PENDING → RUNNING → DONE | FAILED | CANCELLED`, agent-owned after creation).
  Created ONLY by the accept action in the UI. No human approval op → no agent run,
  by construction.
- **Proposal** (any node kind, authored `Actor.Agent(id)`, field `proposed=true`):
  the agent's output, parented under the subject. Proposals never mutate existing
  human-owned fields — they are new child/annotation objects (same field-ownership
  rule the M8 runtime enforces for operators; reuse its enforcement, not a copy).

Accept/reject are human ops (already implemented against the stub in
`feat/m9-hardening`'s web UI): accept = `SetField(proposed=false)` (+ optional
move); reject = `Tombstone`. Both idempotent, both merge like any other op.

## 2. Runtime loop (server, extends M8)

`AgentRuntime(db, service, llm: LlmClient, registry: AgentRegistry)`:

1. Trigger: via the same `OpIngestHook` fan-out M8 uses — react to ops creating
   `agent_task` nodes with `status=PENDING` (plus a startup sweep for tasks left
   `PENDING`/`RUNNING` by a crash; `RUNNING` older than a timeout → `FAILED`).
2. Claim: CAS the status field `PENDING→RUNNING` via one `ingestLocal` op
   (`Actor.Agent`); the op-log's single-writer transaction is the lock.
3. Run: `Agent.run(subject snapshot, tools) → List<ProposalDraft>`. Long-running is
   fine (own coroutine scope, `ZYNC_AGENT_TIMEOUT_MINUTES`, default 30). Tools are
   read-only lookups over the projection + LlmClient calls; NO network fetches in v1.
4. Emit: proposals as an atomic bundle (needs `emitBatch` — see dependencies),
   then `status=DONE`. Failure → `status=FAILED` + `error` field; retry is a human
   re-approval, never automatic (cost tier).
5. Budget: per-task LLM-call fuel (`ZYNC_AGENT_FUEL`, default 20 calls) enforced in
   the runtime, mirroring M8's cascade fuel.

Reference agent (v1): **project-research** — for a project node, propose a comment
node summarizing next actions from the subtree. Deterministic fake in tests.

## 3. Security / threat model deltas

- Agents inherit the operator posture: proposal-only writes, field-ownership
  enforcement by construction, no destructive ops (`Tombstone` forbidden from
  `Actor.Agent` — assert in the runtime AND reject at `ingestLocal`).
- Prompt injection: agent input is user content; output is only ever proposals a
  human reviews. Cap proposal count per run (`ZYNC_AGENT_MAX_PROPOSALS`, default 20).
- Provenance: `agentTaskId` stamped on every proposal (field, not op metadata) so
  the UI can group and bulk-reject a bad run.

## 4. Test plan

- Runtime: approval-creates-run (op → RUNNING → DONE with proposals), crash sweep,
  timeout → FAILED, fuel exhaustion, tombstone-from-agent rejected, proposal cap.
- Flow: operator recommendation → (no run) → human approve → proposals → accept
  keeps content with `proposed=false`; reject tombstones. End-to-end against the
  fake LLM, mirroring `AutoClarifyEndToEndTest`.
- UI: Playwright — proposals render flagged, accept/reject buttons emit the ops
  (the stub producer from this branch is replaced by the real runtime).

## 5. Dependencies / order

1. Merge `feat/m8-operators` (runtime, `LlmClient`, `OpIngestHook`) and
   `feat/m9-hardening` (this branch) into main.
2. `emitBatch` on `OpEmitter`/`ingestLocal` (Codex review #14) — proposals must land

3. Implement `AgentRegistry` + `AgentRuntime` + reference agent + tests (§2, §4).
4. Wire the stubbed UI producer to the real runtime; add the Playwright spec.
5. Acceptance: run the 2026-07-13 runbook items that touch operators/agents on
