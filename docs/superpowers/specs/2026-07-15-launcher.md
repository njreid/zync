# zync as the Android launcher — spec (2026-07-15)

> Status: SPEC, approved direction (decisions taken 2026-07-15): **full HOME
> replacement**, with a Pixel-style **swipe-left combined app + web search** as the
> escape hatch, and the **current GTD context pinned at the top of the screen** as a
> dropdown. Not yet scheduled; sequenced below.
>
> History note: this is NEW design, not restoration — no HOME intent filter ever
> existed in git history. What v0.2 actually had was a 4-action capture bar
> (Text/Voice/Scan/Upload, Font Awesome icons, `ZyncCapture` bridge) whose styling
> survives in the restored custom.css; its capture modes are folded in here.

## Vision

The phone's home screen IS the GTD surface: your inbox/next actions for the current
context, one swipe from everything else. zync stops being an app you open and becomes
the place other apps are launched from.

## Components

### L1 — bottom action bar (native Compose, overlaying the WebView)
Five fixed slots, Font Awesome iconography (v0.2 `fa-icon` style):

| Slot | Action |
|---|---|
| Messages | `Intent.CATEGORY_APP_MESSAGING` (default SMS/RCS app) |
| Calendar | `Intent.CATEGORY_APP_CALENDAR` |
| Context app | The app bound to the CURRENT zync context (see L4); hidden or settings-shortcut when unmapped |
| Phone | `ACTION_DIAL` |
| Capture | Expands the v0.2 capture sub-bar: Text (quick-add via loopback), Voice, Scan, Upload — existing bridge/native flows |

Native (not `:web`) because the slots fire Android intents and query
PackageManager; the WebView shell shrinks by the bar height (insets-aware, like
v0.2's fixed bar + safe-area padding).

### L2 — HOME registration + role
- `LauncherActivity` (or MainActivity alias) gains `HOME` + `DEFAULT` categories;
  `launchMode` stays `singleTask`; pressing home = surface zync, never exit.
- Prompt once with `RoleManager.createRequestRoleIntent(ROLE_HOME)`; a Settings row
  to re-trigger. Graceful as a non-default launcher too (bar still works, L1 stands
  alone).
- Back gesture on the home surface does nothing (launcher convention); WebView
  history navigation only applies off the root view.

### L3 — swipe-left: combined app + web search (the escape hatch)
- Full-screen Compose overlay on horizontal swipe (and a search affordance in the bar
  region): one text field, results as you type —
  - **apps**: `LauncherApps`/PackageManager query over launchable activities, icon
    grid, tap to launch (this is the app drawer);
  - **web**: "Search the web for …" row → `ACTION_WEB_SEARCH`/browser;
  - later: zync tasks/notes in the same results (ContentReadModel search).
- Must be fast and empty-state-useful (recent/all apps grid before typing).

### L4 — context header (`:web`, shared with the browser)
- A pinned pill at the top of every `:web` page: the CURRENT context (e.g.
  `@errands`); tap → dropdown of all contexts + "none". Selection filters
  inbox/tree/lists (context = existing tag model; the read model needs a
  context-filter parameter, which v0.2 had as `context-filter-bar`).
- Selected context is a client concern per surface (cookie/localStorage on browser;
  loopback the same) — NOT synced state, so phone and desktop can sit in different
  contexts.
- The context→app binding used by the bar's "Context app" slot is stored per context
  as a field on the context node (op-log, syncs like everything else); the phone
  resolves it to an installed app at tap time.

### L5 — automatic context switching (later; operators tie-in)
Geofencing (`@errands` near the shops), time-of-day (`@computer` on work hours),
calendar events (meeting → `@agenda:person`). Candidates emit context-switch
suggestions; switching stays local-first (a suggestion chip, not silent). Pairs
naturally with the M8 operator runtime once agents/operators can read calendar/
location signals — spec that integration when L4 exists.

## Non-goals (v1)

- Widgets, wallpaper management, icon packs, notification badges — launcher table
  stakes we deliberately skip; zync's home surface is the task list, not an icon grid
  (the icon grid lives behind swipe-left search).
- Syncing the selected context across devices.

## Testing

- Robolectric: bar intents resolve (package queries stubbed), role prompt requested
  once, back-on-home consumed, context pill filter round-trips via loopback.
- Playwright: context pill dropdown filters the inbox (dev server).
- Runbook (device): set-as-home flow, swipe search launches apps, capture sub-bar
  end-to-end, home gesture behavior.

## Sequencing

1. **L4 context header** — pure `:web` + read-model work, benefits browser users
   immediately, and the bar's context slot depends on it.
2. **L1 bar** (with capture sub-bar) — native, independent of HOME.
3. **L2 HOME + role prompt** — small once L1 exists.
4. **L3 swipe search** — the biggest native piece; HOME without it is livable for a
   day or two but ship them close together.
5. **L5 auto-context** — after operators mature.
