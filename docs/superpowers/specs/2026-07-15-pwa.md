# zync `:web` as a PWA — spec (hold: not yet implemented)

> Status: SPEC (2026-07-15). Approved direction, implementation deliberately held.
> Scope decision: **installability + app chrome, NOT offline content.** The browser is
> an online-only thin client by design (rebuild roadmap "Not doing"); the phone is the
> only offline replica. An offline browser replica would be a re-architecture, not a PWA.

## Goals

- "Install zync" on desktop Chrome/Edge and Android "Add to Home screen": standalone
  window, real icon, no browser chrome.
- Graceful offline behavior: a cached fallback page ("zync needs a connection — the
  phone app works offline"), never a stale inbox.
- Zero effect on the phone WebView (loopback) — the native shell IS the app there.

## Non-goals (explicitly out of scope)

- Offline mutations/queueing in the browser (phone's job).
- Web push notifications (real feature: VAPID keys, subscription store, send path —
  spec separately if wanted).
- iOS-specific polish beyond `apple-touch-icon` (works, but not a target).

## Deliverables

### 1. Manifest (`web` module)
- Route `GET /manifest.webmanifest` (static string or small serializable DTO):
  `name: "zync"`, `short_name: "zync"`, `start_url: "/"`, `display: "standalone"`,
  `theme_color` / `background_color` matching the UI, `icons` (192, 512, + maskable).
- `Layout.kt` head: `<link rel="manifest" href="/manifest.webmanifest">`,
  `<meta name="theme-color">`, `<link rel="apple-touch-icon">`.
- Icons served under `/assets/` from `web/src/commonMain/resources` (same pattern as
  the vendored `datastar.js`). Asset work: 192/512 PNG + maskable variant.

### 2. Service worker (`web` module)
- `GET /sw.js` at root scope. Strategy:
  - navigations: network-only with a pre-cached `/offline` fallback page;
  - static assets (`/assets/*`): cache-first, cache name versioned per release
    (embed the app version; old caches purged on activate).
- Registration from a small **external** script (CSP is `'self'` + eval-for-Datastar;
  no inline), and **only when `location.hostname !== '127.0.0.1'`** — the phone
  loopback must never get a service worker caching UI under the native shell.

### 3. Session persistence (server; review finding #7 — pairs with this work)
An installed app that logs out on every server deploy feels broken. Move `SessionStore`
to SQLite: store sha256(token) + expiry (new table ⇒ next `.sqm` migration + version
assertions bump), purge on mint, cap active sessions. In-memory behavior stays for the
loopback/dev path.

### 4. Tests
- Playwright (dev server): manifest served + parseable; SW registers and controls the
  page; `/offline` fallback renders when the SW serves it.
- Loopback CSP spec: assert NO service worker registration on `127.0.0.1`.
- Server: sessions survive a simulated restart (new store, same DB).

## Sequencing when implemented

1. Session persistence (schema v(N+1) migration + tests).
2. Manifest + icons + head tags.
3. Service worker + offline fallback + registration guard.
4. Playwright coverage; acceptance-runbook additions (install on Android/desktop,
   offline fallback check, sessions survive a deploy).

Estimated effort: ~half a day including the session work.
